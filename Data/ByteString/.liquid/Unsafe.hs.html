239: -- | /O(1) construction/ Use a 'ByteString' with a function requiring a
240: -- 'CString'.
241: --
242: -- This function does zero copying, and merely unwraps a 'ByteString' to
243: -- appear as a 'CString'. It is /unsafe/ in two ways:
244: --
245: -- * After calling this function the 'CString' shares the underlying
246: -- byte buffer with the original 'ByteString'. Thus modifying the
247: -- 'CString', either in C, or using poke, will cause the contents of the
248: -- 'ByteString' to change, breaking referential transparency. Other
249: -- 'ByteString's created by sharing (such as those produced via 'take'
250: -- or 'drop') will also reflect these changes. Modifying the 'CString'
251: -- will break referential transparency. To avoid this, use
252: -- 'Data.ByteString.useAsCString', which makes a copy of the original 'ByteString'.
253: --
254: -- * 'CString's are often passed to functions that require them to be
255: -- null-terminated. If the original 'ByteString' wasn't null terminated,
256: -- neither will the 'CString' be. It is the programmers responsibility
257: -- to guarantee that the 'ByteString' is indeed null terminated. If in
258: -- doubt, use 'Data.ByteString.useAsCString'.
259: --
260: -- * The memory may freed at any point after the subcomputation
261: -- terminates, so the pointer to the storage must *not* be used
262: -- after this.
263: --
